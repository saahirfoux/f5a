Overview

Personas vs roles: session personas describe in-room behavior; account roles describe RBAC and menu visibility. One signed-in user may be Host in a session and Manager on their account.

Three tiers in this document

  • Tier 1 Session personas (2) — drive Storybook and core flows: Host, Participant
  • Tier 2 Account personas (5) — menu, gating, admin surfaces: Moderator, Manager, Admin, Reporter, Super Admin
  • Reference Market archetypes (5) — who Host often is in the wild; copy/messaging only, not separate UI trees

Canonical sources: f5m/documentation/foundational/fist_of_five_personas_v4.md, f5m/documentation/implementation/canonical_contract_pack.md, f5a/ONBOARDING_OBJECTIVE_REFINEMENT.md, f5a/documentation/ux/f5a_mobile_ux_requirements.md.

Tier 1 — Session personas

Primary Storybook and route drivers. Vote scale: 1–5 only.

1. Host Session

Session Controller · HOST role · f5a native primary user · Life scenarios traceability

Who they are
One persona, many life modes — same signed-in host, different room context:
  • Social hurry host — party, pizza decision, multitasking (Story A)
  • Daily Scrum Master — standup vibe check, identify low voter by name (Story B)
  • Auditorium presenter — anonymous room, projector results (Story C)
  • Async creator — YouTuber, many sessions, link-in-bio distribution (Story D)
Also maps to market archetypes (see Market archetypes) but remains a single Host persona. Source stories: life-scenarios-for-host.md.
Job to be done
  • Go live in seconds — party, standup; one control → QR + shareable link
  • See who voted what when identified — follow up on low scores (John = 3)
  • Show anonymous aggregate to the room when needed — projector / public results link
  • Manage many sessions over time — weekly videos, list with activity badges
  • Answer “How confident is this audience, right now?” in under a minute
Per-person results and spread must never be buried for identified sessions.
Session presets (not personas)
PresetLife modeFlags / behaviorPriority
Team standup Story B Identified; per-session QR; host sees names; pre-share link via email v1
Party quick Story A Default/skip topic UI; glanceable aggregate; optional push (v1.1); partial turnout OK v1 + v1.1
All-hands anonymous Story C Anonymous-to-room; public live results URL; multi-topic; presenter web view v1.1
Creator / link-in-bio Story D Anonymous web voters; session list CRUD; unread vote badges v1.1
Presets are session configuration templates — not separate personas.
Platform & entry
f5a native iOS/Android (primary). Expo web at app.fistoffive.co secondary. Presenter public results may be web-only (Story C). Signs in via Supabase; guest-host behind feature flag (short-term only).
Key flows
  1. Land on home → dominant Go live (or Resume if session active)
  2. Go live → optional skip topic UI → default topic created → share via HostShareCard
  3. Enter session before realtime room (v4 invariant); v1.1 may offer auto-enter for ad-hoc
  4. Live host view: LiveVoteRoster + SpreadSummaryBar; return via push or resume
  5. Presenter (v1.1): copy public results link → open on laptop → share screen to room
  6. Creator (v1.1): dashboard session list with badges, rename, delete
Routes
/host/<session_id> · home shell · session list (v1.1) · menu · in-session settings
Success metrics
Primary: time-to-first-session. Secondary: decision made without reopening app (Story A); identified low voter within first minute of standup (Story B).
Storybook priority
High (v1) — GoLiveCTA, LiveVoteRoster, SpreadSummaryBar, HostShareCard, TopicHeader, RealtimeBanner. v1.1 — PublicResultsLinkCard, SessionListRow, SessionSummaryBar. Build order: Storybook priority review. Features: feature roadmap.

2. Participant Session

Session Visitor · no account role required · guest-first

Who they are
Someone in the audience who received a QR code or invite link. Does not need the full app; prioritizes speed and low friction.
Job to be done
Join the session and cast a 1–5 confidence vote on the current topic. Understand when voting is open, closed, or the session has ended.
Platform & entry
iOS App Clip (vote-first, minimal chrome) · web via link · full f5a app optional. No manual session code entry — QR or deep link only.
Auth variants
  • require_auth = false, require_name = false → anonymous (user_id null)
  • require_name = true → display name prompt
  • require_auth = true → sign in or “Open full app to sign in”
These are session flags, not separate personas.
Key flows
  1. Open /session/<session_id> from link or QR
  2. If SCHEDULED pre-start → see start time, voting disabled
  3. If ACTIVE → vote 1–5 via VoteControl
  4. If ENDED / EXPIRED → SessionStateFullScreen “Session is over”
  5. Optional “Get full app” CTA — must not block voting
Out of scope (App Clip)
Create session, invites, roles, billing, host takeover, multi-session browse
Storybook priority
High — VoteControl, SessionStateFullScreen, TopicHeader, RealtimeBanner, App Clip compact layout

Tier 2 — Account personas

RBAC-driven menu and gated actions. Lower Storybook priority than Tier 1.

3. Moderator Account

MODERATOR · account-wide session oversight

Who they are
Account lead or agile program manager who oversees multiple facilitators and sessions across the organization.
Job to be done
Assign hosts to sessions, view and edit all account sessions, ensure sessions run smoothly across teams.
Includes
All HOST capabilities across the account (not only own sessions).
Key surfaces
Session list (all hosts), host assignment, session edit — often desktop-first; mobile secondary.
Storybook priority
Low — defer until Tier 1 session loop ships

4. Manager Account

MANAGER · people & permissions

Who they are
Team or department lead responsible for who has access to the Fist of Five account.
Job to be done
Invite colleagues, assign roles, disable users. Ensure the right people can host and report.
Key surfaces
Menu → Invites, Role management (Managers+ per UX spec).
Gating
GatedButton when subscription seat limits apply.
Storybook priority
Medium-low — Invites list, role change UI, GatedButton states

5. Admin Account

ADMIN · account owner · billing

Who they are
Account owner or IT buyer who purchased the subscription and owns billing relationship.
Job to be done
Manage plan, usage, upgrades. Full account control including all Manager capabilities.
Key surfaces
Menu → Subscription / billing (ADMIN only; others see hidden or “Contact your admin”).
Storybook priority
Low — subscription screen deferred; Stripe integration not in v1 auth scope

6. Reporter Account

REPORTER · read-only analytics

Who they are
Stakeholder who needs visibility into confidence trends across sessions without running or editing them.
Job to be done
View all account sessions, results, topics, and participants. Export or review historical spread.
Key surfaces
Analytics / reporting (secondary on phone per UX spec). No edit permissions.
Storybook priority
Low — defer; likely web-first or tablet layout

7. Super Admin Account

SUPER_ADMIN · platform operator

Who they are
Platform owner (you) — internal ops, not a customer persona.
Job to be done
Platform-wide analytics, account oversight, support and health monitoring.
Key surfaces
Internal tools; rarely on mobile. Not a v1 Storybook target.
Storybook priority
Defer
